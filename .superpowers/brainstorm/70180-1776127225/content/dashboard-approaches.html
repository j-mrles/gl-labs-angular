Staff Dashboard — Which approach for the tool cards?

All 4 tool cards (Contact Inquiries, Site Content, Analytics, Settings) currently go nowhere. Pick an approach to fix them.

A

In-page panels (no new routes)

Clicking a tool card expands or slides in a section within the dashboard page itself — like a drawer or tab. No new URLs. All content lives on /staff-dashboard.

Pros

  • Zero new routes to create
  • Fast — no page load
  • Simple to implement

Cons

  • Can feel cramped if content grows
  • No bookmarkable URLs per tool
B

Sub-routes (dedicated pages)

Each tool gets its own route and component: /staff-dashboard/inquiries, /staff-dashboard/content, /staff-dashboard/analytics, /staff-dashboard/settings. Most scalable long-term.

Pros

  • Clean URL per tool
  • Fully scalable — each page can grow independently
  • Back button works naturally

Cons

  • 4 new components + routes needed
  • More code upfront
C

External integrations (link out)

Each card links directly to the real tool already in use — Formspree for contact inquiries, Firebase console for analytics, etc. Minimal code, uses existing infrastructure.

Pros

  • No new code — just update the URL
  • Uses real tools you already have
  • Instant fix

Cons

  • Sends staff to external sites
  • Looks less polished / integrated
  • Requires knowing the right URLs

My recommendation

Option B (sub-routes) — the app already has a clean route structure and an auth guard on /staff-dashboard. Sub-routes extend that naturally, give each tool room to grow, and match the architecture pattern already in use. The upfront cost is low: 4 small components with placeholder content that can be built out over time.
